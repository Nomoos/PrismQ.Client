# Root Folder Structure

## Files and Folders in Root Directory

### Files in Root

1. **`.gitignore`** - Git configuration file
   - **Why in root:** Required by Git at repository root
   - **Purpose:** Specifies which files Git should ignore

2. **`.nvmrc`** - Node Version Manager configuration
   - **Why in root:** Development tools look for this at repository root
   - **Purpose:** Specifies the Node.js version for the project

3. **`README.md`** - Project documentation
   - **Why in root:** Standard location for project entry point
   - **Purpose:** First file users see, provides project overview and getting started guide

4. **`VERSION`** - Version metadata
   - **Why in root:** Deployment metadata needed at root level
   - **Purpose:** Tracks current version of the application

### Folders in Root

1. **`.git/`** - Git repository data
   - **Why in root:** Required by Git
   - **Purpose:** Contains all version control history and configuration

2. **`.vite/`** - Build cache (temporary)
   - **Why in root:** Generated by build tools
   - **Purpose:** Vite build cache (should be in .gitignore)
   - **Note:** Not committed to repository

3. **`_meta/`** - All meta-information and documentation
   - **Why in root:** Central location for all non-deployment files
   - **Contents:**
     - `_meta/docs/` - All documentation files
     - `_meta/examples/` - Example code and configurations
     - `_meta/issues/` - Issue tracking
     - `_meta/tests/` - Test documentation and load tests
     - `_meta/_scripts/` - Utility scripts

4. **`Backend/`** - Backend deployment code
   - **Why in root:** Main module for backend implementation
   - **Contents:**
     - `Backend/TaskManager/` - TaskManager API implementation
   - **Purpose:** Contains backend code that gets deployed to production

5. **`Frontend/`** - Frontend deployment code
   - **Why in root:** Main module for frontend implementation
   - **Purpose:** Contains frontend code that gets deployed to production

6. **`playwright.config.ts`** - Playwright test configuration
   - **Why in root:** Testing framework looks for config at root
   - **Purpose:** Configuration for end-to-end tests

## Principles

### What Stays in Root
- **Configuration files** required by tools (`.gitignore`, `.nvmrc`)
- **Essential project files** (README.md, VERSION)
- **Git repository** (`.git/`)

### What Goes in `_meta/`
- **All documentation** (guides, summaries, plans)
- **Example code** (not for deployment)
- **Issue tracking**
- **Test documentation**
- **Utility scripts**

### What Goes at Root as Modules
- **Backend/** - Backend deployment code (modules with Uppercase names)
- **Frontend/** - Frontend deployment code (modules with Uppercase names)
- **Module-specific _meta/** (e.g., `Backend/TaskManager/_meta/`) contains module-specific documentation and meta-information

## Benefits

1. **Clear Separation**
   - Deployment code (Backend/, Frontend/) vs documentation (_meta/) clearly separated
   - Module-level organization with each module having its own _meta/
   - Easy to deploy only what's needed

2. **Clean Root**
   - Minimal clutter at root level
   - Only essential files and main modules
   - Easy to navigate

3. **Scalable**
   - New modules go at root level (e.g., Backend/, Frontend/)
   - New module docs go in module's _meta/ folder
   - Pattern is consistent across all modules

4. **Deployment-Ready**
   - Backend/ and Frontend/ contain everything needed for deployment
   - No accidental deployment of docs or examples
   - Clear separation between code and meta-information
